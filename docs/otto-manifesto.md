# Otto Manifesto

## What Otto Is

Otto is a Pi-native operating layer for BMAD + td.

It is not a generic agent shell. It is not a thin command wrapper. It is a system for running meaningful software-delivery loops with tighter context, stronger judgment, and better continuity than ad hoc agent driving.

Otto exists for developer AI power users who already live inside coding agents and want the next real jump in velocity.

## Who It Is For

Otto is for people who:

- spend much of the day inside AI coding tools
- already understand agentic workflows and chained loops
- care more about real leverage than novelty
- want a system that can carry work forward with less babysitting

Otto is not optimized first for beginners. Its first job is to feel indispensable to expert operators.

## The Core Bet

The product bet is simple:

- if Otto makes good next-step decisions consistently
- and keeps the working context sharp
- and validates against real product behavior
- then it can drain meaningful queues with surprisingly little babysitting

That is the magic.

The visible win is queue drain.

The underlying win is trustworthy judgment over a long-running process.

## Otto's DNA

Otto should remain:

- mixed-autonomy, not autonomy-maximalist
- closed-loop, not one-shot
- session-aware, not context-bloated
- human-steerable, not human-absent
- reality-maximalist, not artifact-maximalist

Its core loop is:

1. prepare the workspace
2. execute the highest-value next step
3. preserve review separation and context sharpness
4. validate against the PRD and real runtime behavior
5. reopen gaps as actionable work

## What Otto Must Care About Most

When goals conflict, Otto should optimize in this order:

1. decision quality
2. traceability and confidence
3. low-friction operator flow
4. raw speed

Speed matters. But speed without judgment creates fake wins.

## Product Truth Over Artifact Truth

Otto should never confuse workflow completion with product completion.

Stories can be done. Tasks can be closed. Tests can pass. Outputs can look plausible. And the product can still fail its core requirement.

That is one of the most dangerous failure modes in agentic delivery.

Otto must be designed to detect it, not decorate it.

The standard is product truth:

- does the real core loop work?
- does the runtime behavior match the promise?
- does the delivered surface satisfy the PRD where it actually matters?

## Evidence Must Be Real

Otto should trust signals in this order:

1. real runtime behavior
2. direct validation against PRD core-loop assertions
3. human review of the working product
4. automated tests and checks
5. workflow and artifact completion signals

Tests matter, but they can be spoofed.

For web UI, real browser validation and vision-based review matter.
For CLI systems, real commands and outputs matter.
For interactive tools, live state and observable behavior matter.

Approval should mean reality was checked.

## Tight Context Is A Feature

Otto should be opinionated about keeping context tight.

Long, blurry context windows make models slower, weaker, and more likely to drift. td + BMAD are not only workflow primitives; they are context-management primitives.

So Otto should prefer:

- fresh-session iteration
- minimal continuity state
- explicit current td framing
- compaction as fallback, not default operating model

Context freshness per decision is part of the product advantage.

## The Human Role

Otto should support different autonomy boundaries for different projects.

Default serious-work posture:

- humans stay in the loop for planning pivots
- humans stay in the loop for consequential approval decisions

But exploratory work may choose lighter gates. Otto should support that without pretending all work has the same risk profile.

The point is not one fixed autonomy philosophy.
The point is governed autonomy with smart defaults.

## What Otto Should Be Opinionated About

Otto should have strong defaults for:

- validation and approval standards
- next-step prioritization
- session and continuity behavior

These are the levers that most strongly determine trust.

## What Otto Should Become

Otto should evolve in stages:

1. best-in-class orchestration and operator UX for BMAD + td
2. stronger judgment layer: evidence quality, drift detection, runtime truth checks, confidence
3. deeper workflow semantics where direct ownership increases reliability

The goal is not to replace BMAD and td prematurely.
The goal is to become the best system for driving them, then grow into the places where stronger product intelligence is needed.

## The Standard

If Otto succeeds, an expert user should feel this:

- I trust its next move more often than not.
- I do not need to keep reloading the whole situation into context.
- When it says work is real, it has evidence.
- When it sees drift, it corrects course.
- When it runs for a while, it compounds my output instead of compounding confusion.

That is the bar.
